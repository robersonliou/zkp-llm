ZKP Primer — Interactive → Non-interactive

Interactive Proof System (GMR 1985)

  • Completeness — honest prover always convinces verifier
  • Soundness — cheating prover fails with high probability
  • Zero-knowledge — verifier learns nothing beyond the statement

The limitation

  • Prover and verifier must be online simultaneously
  • Verifier sends fresh random challenges each round
  • Not suitable for blockchain or offline verification

Fiat-Shamir Heuristic — the bridge

Replace the verifier's random challenge with a cryptographic hash: $$e = H(\text{transcript so far})$$ Prover computes the challenge deterministically → non-interactive

Result: NIZK — Non-Interactive Zero-Knowledge

  • Proof is a single message; verifier works offline
  • All modern SNARK / STARK / zkVM stand on this bridge
  • Course link: Week 7 Schnorr $e = H(C)$ → Fiat-Shamir transcript
Sigma protocol pattern: Commit → Challenge → Response
Fiat-Shamir collapses the interactive round into a hash

Lookup Arguments — the 2024 Breakthrough

Why lookups?

  • Non-arithmetic ops are circuit-unfriendly:
    Softmax, GELU, SwiGLU, range checks
  • Encoding them as arithmetic constraints → huge circuit size
  • Solution: prove "every value I used is in this pre-committed table"

Evolution timeline

  • Plookup (Gabizon-Williamson 2020) — first practical lookup PIOP using sorted multisets
  • logUp — log-derivative formulation, cleaner algebraic structure
  • cq / cached quotients — preprocess table, amortize over many queries
  • tlookup (zkLLM 2024) — tensorized, batch entire attention head at once

Lookup core idea

Table $T = \{(x, f(x))\}$. Prover commits to witness $W$.
Proves: every row of $W$ appears in $T$
Using: grand product / log-derivative sum check

Why this unifies our three papers

  • zkLLM — tlookup for Softmax / attention
  • zkRAG — hybrid lookup for neighbor lists
  • EZKL — Halo2 lookup for non-linear ops
Lookup arguments are the common thread connecting all three backends in this project

Folding Schemes — Recursive Proof Compression

The problem

  • Proving a long trace (e.g. 1M mcycles) in one shot → proof too large
  • Incrementally Verifiable Computation (IVC): prove step-by-step, accumulate

Folding: fold multiple instances into one

Two R1CS instances $(x_1, w_1)$ and $(x_2, w_2)$
→ folded instance $(x, w)$ with smaller size
Recurse until a single compact proof remains

Lineage

  • Nova (Microsoft 2022) — first practical folding scheme
  • SuperNova — multi-circuit folding
  • HyperNova (ePrint 2023/573) — CCS-based, optimal

Tooling

  • microsoft/Nova — reference implementation
  • privacy-scaling-explorations/sonobe — production library
DEAAP future direction:
Use folding to compress N Cartesi step proofs into a single proof
→ amortize prover cost across the entire LLM inference trace
Not implemented in current PoC — identified as the primary future optimization

zkLLM — Motivation & tlookup

Core problem

  • LLM inference involves non-arithmetic operations:
    Softmax, GELU, SwiGLU
  • Traditional SNARKs encode these as giant polynomial constraints → impractical

tlookup: tensorized lookup argument

  • Batch an entire attention head or FFN block in a single PIOP
  • All heads share one small table

Softmax trick — shift invariance

$$\text{softmax}(s_{ij}) = \text{softmax}(s_{ij} - \max_j s_{ij})$$ Subtract row-max → values fit in a small range → small table

Base-$b$ digit decomposition

$$x = \sum_k d_k \cdot b^k, \quad \exp(x) = \prod_k \exp(d_k b^k)$$ Each digit $d_k$ looks up a table of size $O(b)$
Total table size: $O(b \cdot \log_b M)$

Why is HNSW hard to prove?

RISC0 Step Prover Workflow

Four-step pipeline

# Step 1: Create machine snapshot
cartesi-machine ... --store=./snapshot

# Step 2: Collect state root hashes per mcycle
cartesi-machine --load=./snapshot \
    --collect-mcycle-root-hashes=./roots.bin

# Step 3: Prove single mcycle step transition
./cartesi-risc0-guest-step-prover \
    --pre-state=roots[t].bin \
    --post-state=roots[t+1].bin \
    --output=step_proof.bin

# Step 4: Verify on-chain via Solidity
cast call $VERIFIER "verify(...)" $proof $PRE $POST

What is being proven?

The state transition:
$$\text{pre\_root} \xrightarrow{\text{RISC-V exec}} \text{post\_root}$$ is a valid single Cartesi machine step

Proof artifacts

  • step_proof.bin — RISC0 STARK receipt
  • step.public.json — {pre_root, post_root, mcycle}
  • Both feed directly into StepVerifier.sol

  • EZKL calibrate is very sensitive to input range — always use representative samples from the actual data distribution
  • ONNX opset 11–17 works well; opset 18/19 has reduced EZKL support

  • RISC0 toolchain first-time compile: ~30 minutes (cargo install dominated)
  • RISC0_DEV_MODE=1 skips witness generation → dev receipts are not real STARK proofs
  • Full STARK proof requires ≥16 GB RAM — not practical on a laptop

Q & A

Expected questions

  • Why not prove the entire pipeline with one zkVM?
    Orders of magnitude slower — see benchmark slide. Specialized per-layer provers are 100–1000× faster.
  • What is the difference between tlookup and Plookup?
    Plookup works element-wise; tlookup is tensorized — it batches an entire attention head in one PIOP using a shared table.
  • How does Cartesi v0.20.0 compare to OP Stack / Arbitrum Stylus?
    Cartesi uses a full Linux VM (RISC-V), enabling richer off-chain compute. v0.20.0 adds ZK fallback that OP Stack lacks natively.
